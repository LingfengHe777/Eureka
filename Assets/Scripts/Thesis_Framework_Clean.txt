毕业设计（论文）
题目：基于Unity3D的Roguelike类PC游戏的设计与实现
作者：本人

摘要
随着独立游戏市场的发展，融合Roguelike机制与实时生存玩法的游戏逐渐成为动作游戏的重要类型。这类作品强调高密度战斗、随机成长和高复玩性，同时也对系统架构、数值平衡和性能稳定性提出了更高要求。本文以本人基于Unity3D开发的2D俯视角游戏原型为研究对象，围绕完整玩法循环、模块化工程实现、参数调优与运行优化展开系统研究。项目以战斗、资源结算、升级和商店决策为主循环，构建了会话管理、状态控制、波次调度、敌人生成、玩家成长、武器道具、难度策略与对象池优化等核心模块，并通过配置驱动和策略解耦实现高可维护性。测试结果表明，原型已经形成完整可复玩的核心闭环，难度与经济节奏具备较好可解释性，在高同屏压力下运行稳定性得到明显改善。本文实践说明，在中小规模开发条件下，采用配置化与模块化结合的工程路线，能够有效支撑Roguelike类生存游戏的设计与实现，并为后续扩展敌人行为、地图机制和自动化调参提供基础。

关键词：Unity3D；Roguelike；波次系统；配置驱动；策略模式；数值平衡；性能优化

ABSTRACT
With the growth of the indie game market, real time survival games combined with Roguelike mechanics have become a key direction in action game design. This thesis studies a 2D top down Unity3D prototype developed independently by the author. The project is built around a complete loop of combat, resource settlement, upgrade, and store decision. It implements session management, game state control, wave scheduling, enemy spawning, player progression, weapon and item systems, difficulty strategies, and pooling based optimization. Results from iterative playtests and profiling indicate that the prototype forms a complete playable loop, supports explainable balancing, and maintains improved runtime stability under high enemy density.

第1章 绪论

1.1 研究背景与意义
Roguelike与Roguelite在独立游戏领域持续活跃，类幸存者分支通过低操作门槛和高策略密度建立了广泛用户基础。与传统线性关卡游戏相比，该类型更依赖系统循环质量。玩家在战斗中获得资源，在阶段间进行成长决策，再将构筑结果反馈到下一轮战斗，循环质量直接决定体验强度与复玩意愿。对个人开发者而言，难点不仅是实现功能，更在于在有限时间和人力下建立可维护、可扩展、可调试的工程结构。本文的研究意义在于将玩法设计、软件工程、数值设计和性能优化统一到同一可验证流程中，形成可复用的实现方法。

1.2 国内外研究现状
国外在Roguelike设计与工程实践上积累较早，普遍强调简单操作与深度决策并存，且在高同屏场景下倾向采用轻量行为模型与对象池等稳定手段。国内近年在课程实践和开发社区中也积累了大量经验，常见成果集中在关卡随机、道具系统、波次生成和基础平衡，但完整覆盖玩法闭环、工程结构、性能验证与测试分析的一体化案例仍较少。本文基于现有研究与实践经验，面向个人开发规模给出完整的系统实现与验证路径。

1.3 研究目标与研究内容
本文目标是在Unity3D中完成一款具备完整核心循环的Roguelike类PC游戏原型，并建立可扩展、可调参与可优化的工程化框架。研究内容包括会话与状态管理、波次与刷怪系统、敌人行为系统、玩家成长系统、武器与道具系统、商店与升级决策系统、难度与经济策略系统，以及对象池和资源生命周期优化。除功能实现外，本文还通过测试与实验评估系统稳定性和可解释性。

1.4 研究方法与技术路线
本文采用迭代开发与数据复盘结合的方法。首先完成需求分析和架构拆分，再实现最小可玩闭环，在此基础上逐步补齐核心模块，最后通过多轮测试进行数值和性能调优。技术上采用配置驱动组织数据，采用策略模式管理公式，采用组件化方式组织运行逻辑。该路线能够降低耦合并提升后续维护效率。

1.5 论文结构安排
全文共七章。第一章阐述背景、现状、目标和方法。第二章进行需求分析并给出总体架构。第三章详细说明核心系统实现。第四章介绍实验设计与结果分析。第五章讨论性能优化。第六章呈现系统测试与评估。第七章总结全文并提出后续工作方向。

第2章 需求分析与总体架构设计

2.1 需求分析
本项目属于系统驱动型游戏，核心任务是保证循环闭合和节奏稳定。功能上需要覆盖开始场景的角色与模式选择、战斗场景的波次推进、经验与金币成长、商店与升级决策、胜败结算与无尽分支。非功能上需要满足高同屏压力下的性能稳定、模块职责清晰的可维护性、面向配置和策略的可扩展性，以及关键流程可复现可验证的可测试性。

2.2 核心玩法循环建模
系统核心循环由战斗、结算、升级或商店决策、下一波战斗构成。战斗阶段负责资源产出，决策阶段负责资源转换。生命值、经验值、金币、武器槽位和道具效果是循环中的关键变量。击杀行为带来资源增长，资源增长触发成长选择，成长选择改变属性结构，属性变化再次影响下一波战斗效率。

2.3 总体架构设计
架构设计采用分层思想。表现层承担界面与反馈，业务层承担流程执行，数据层承担静态配置，策略层承担公式计算，运行支撑层承担会话、上下文和对象复用。`GameMgr`作为流程主控负责状态切换，`GameContext`负责依赖收口，`WaveManager`和`SpawnManager`负责战斗节奏与生成执行，玩家模块负责成长和战斗行为。

2.4 关键流程设计
开局流程从会话读取开始，完成全局配置加载、玩家实例化和依赖注册后进入运行状态。单波流程围绕波配置驱动刷怪协程，计时结束后执行停刷和清场。波间流程先处理升级选择，再进入商店决策。终局流程根据普通模式或无尽模式分支进行对应结算。

2.5 数据驱动与配置规范
项目将模式、波次、敌人、武器、道具、升级需求等规则放入ScriptableObject配置，通过统一校验保证字段完整性与数值合理性。难度倍率、刷怪倍率、商品价格、刷新成本、商品抽取和购买判定等公式由策略资产独立管理。业务代码只负责流程调用，不直接持有复杂公式，从而降低回归风险并提升调参效率。

2.6 本章小结
本章明确了项目的需求边界、循环模型、分层架构和关键流程，建立了从目标到实现的统一约束，为后续系统实现与实验验证提供基础。

第3章 核心系统设计与实现

3.1 会话与状态管理
`GameSessionManager`负责跨场景会话保存，确保开始场景选择结果能够稳定传递到战斗场景。`GameMgr`负责游戏状态管理和时间尺度同步，统一控制等待、进行、暂停、胜利和失败状态。`GameContext`负责注册并提供局内关键服务，减少分散查找和隐式依赖。

3.2 波次与刷怪系统
`WaveConfig`定义波次时长、生成点数量、普通怪权重、特殊怪延迟与强度曲线。`WaveManager`负责波次调度、计时控制、波间流程衔接和无尽扩展。`SpawnManager`负责敌人预加载、生成点采样、预警表现和实例生成。系统使用会话标识隔离机制，避免旧协程污染新波次。

3.3 敌人系统与轻量行为实现
`Enemy`抽象基类统一处理生命、受伤、死亡、掉落、倍率应用和池化重置。`EnemyMelee`实现追击与近战攻击时序，`EnemyHarpoonFish`实现远程投掷行为。系统采用分离力计算减少敌人重叠，使围攻形态更自然且可读。

3.4 玩家成长与资源系统
`PlayerMovement`负责移动输入与位移控制，`StatHandler`负责属性统一计算与上下限钳制，`ExperienceManager`与`LevelSystem`负责经验累计和升级触发，`CoinManager`负责金币收支与事件广播。资源系统与成长系统通过事件串联，确保击杀收益、升级反馈和商店消费形成一致闭环。

3.5 武器与道具系统
`WeaponManager`负责槽位管理、武器添加与移除、合成与重排，`WeaponController`负责自动瞄准、冷却控制和攻击执行。近战命中与远程投射物在表现层与结算层保持一致。道具系统采用`ItemConfig`与`EffectSO`分离设计，`EffectManager`负责效果叠加和移除生命周期。

3.6 商店与升级决策系统
`UpgradePanel`提供升级选项，`StorePanel`提供刷新、锁定、购买、出售和继续流程，`PostWavesChoicePanel`承接关键波间决策。购买判定通过策略统一返回结果与原因，避免规则散落在界面逻辑中。波间流程采用先升级后商店的顺序，使玩家在做消费决策时拥有更明确的属性信息和收益判断。

3.7 难度与经济策略系统
`DifficultySystem`根据模式与波次上下文统一输出敌人和经济倍率。ValueSystem中的各类策略分别负责敌人缩放、刷怪倍率、商品价格、刷新成本、商品抽取和购买规则。策略化的核心价值在于将公式从业务代码中剥离，使调参更集中、回滚更安全、验证更直接。

3.8 本章小结
本章完成了核心系统的结构与实现说明。各模块在职责边界清晰的前提下形成协作闭环，既满足当前可玩需求，也保留了后续扩展空间。

第4章 数值设计、实验方案与结果分析

4.1 实验目标与评价指标
实验目标包括难度合理性、成长连续性、经济可决策性和性能可承载性。核心指标包括平均生存时长、通关率、无尽轮次、升级频率、金币结余、刷新次数、高阶商品购买率、单位时间击杀和受击频率。

4.2 实验环境与采样方法
实验在统一版本与场景条件下进行，采用日志记录、手工复盘和Profiler采样结合的方式采集数据。每组参数进行多轮测试并剔除异常样本后统计均值与波动区间，确保结论具有可重复性。

4.3 参数组设计
参数调整分为波次节奏、敌人倍率、商店经济和升级曲线四组。波次节奏关注生成间隔和场上上限，敌人倍率关注生命和伤害增长，商店经济关注价格与刷新成本，升级曲线关注等级推进速度。

4.4 结果分析
实验结果显示，当前参数下难度曲线呈阶段性递进，前期用于学习和建立认知，中期强化挑战，后期强调构筑质量。成长反馈保持连续，刷新行为具有机会成本，商品购买分布随波次推进呈合理变化。对前期过压与中期滚雪球等异常局进行针对性微调后，极端结果比例明显下降，整体稳定性提升。

4.5 本章小结
本章通过分组实验和数据对照验证了参数体系的有效性，证明策略化调参方法能够支持可解释、可追踪和可复验的平衡迭代。

第5章 性能优化与工程化验证

5.1 性能问题定位
项目早期主要瓶颈表现为频繁实例化引发的垃圾回收抖动、波次切换后残留对象干扰、首次资源加载引发卡顿。通过Profiler定位脚本热点和分配峰值后，优化工作围绕对象生命周期和资源加载路径展开。

5.2 对象池优化实现
`GameObjectPoolManager`实现按预制体分类缓存、借出追踪和统一回收，减少重复创建与销毁。波次结束后进行统一回收，保证下一波在干净状态下启动。该机制显著降低了分配波动并平滑帧时间曲线。

5.3 资源加载与生命周期治理
`SpawnManager`在开局阶段完成关键敌人资源预加载，运行期优先命中缓存。界面资源采用按需加载和释放策略。协程与事件订阅在对象禁用或销毁阶段统一清理，避免悬挂任务和重复回调引发隐性性能问题。

5.4 优化效果评估
优化后在高压场景下平均帧率提高，低帧占比下降，垃圾回收峰值减少，波次切换更稳定。尽管仍有进一步优化空间，但当前结果已满足原型阶段的稳定运行与持续迭代需求。

5.5 本章小结
性能优化并非单点技巧，而是流程管理、对象生命周期和资源策略协同的结果。本章验证了工程化优化路径在本项目中的有效性。

第6章 系统测试与结果评估

6.1 测试策略
测试采用功能验证、边界验证、异常验证和压力验证并行推进。每类测试均包含前置条件、操作步骤、预期结果和实际结果，保证测试过程可复现、可审查和可追踪。

6.3 边界与异常测试结果
边界与异常测试覆盖空配置、余额不足、快速连续操作、依赖晚到、加载失败等场景。通过状态守卫、空值检查、超时等待和降级分支，系统在异常情况下保持可控，不会因单点错误导致整体流程崩溃。

6.4 压力测试结果
在提升同屏敌人数和投射频率的压力场景下，系统保持可玩性，主要瓶颈集中在物理与碰撞开销。该结果与性能分析结论一致，说明后续优化应继续聚焦碰撞层管理、更新节流和高频对象行为开销。

6.5 本章小结
测试结果验证了系统在当前规模下的可靠性与工程可维护性，为论文结论提供了可量化支撑。

第1章 绪论

1.1 研究背景与意义
Roguelike与Roguelite在独立游戏领域持续活跃，类幸存者分支则通过低操作门槛和高策略密度建立了广泛用户基础。与传统线性关卡游戏相比，该类型更依赖系统循环质量。玩家在战斗中获得资源，在阶段间进行成长决策，再将构筑结果反馈到下一轮战斗，循环质量直接决定体验强度与复玩意愿。对个人开发者而言，难点不仅是实现功能，更在于在有限时间和人力下建立可维护、可扩展、可调试的工程结构。本文的研究意义在于将玩法设计、软件工程、数值设计和性能优化统一到同一可验证流程中，形成可复用的实现方法。

1.2 国内外研究现状
国外在Roguelike设计与工程实践上积累较早，普遍强调简单操作与深度决策并存，且在高同屏场景下倾向采用轻量行为模型与对象池等稳定手段。国内近年在课程实践和开发社区中也积累了大量经验，常见成果集中在关卡随机、道具系统、波次生成和基础平衡，但完整覆盖玩法闭环、工程结构、性能验证与测试分析的一体化案例仍较少。本文基于现有研究与实践经验，面向个人开发规模给出完整的系统实现与验证路径。

1.3 研究目标与研究内容
本文目标是在Unity3D中完成一款具备完整核心循环的Roguelike类PC游戏原型，并建立可扩展、可调参与可优化的工程化框架。研究内容包括会话与状态管理、波次与刷怪系统、敌人行为系统、玩家成长系统、武器与道具系统、商店与升级决策系统、难度与经济策略系统，以及对象池和资源生命周期优化。除功能实现外，本文还通过测试与实验评估系统稳定性和可解释性。

1.4 研究方法与技术路线
本文采用迭代开发与数据复盘结合的方法。首先完成需求分析和架构拆分，再实现最小可玩闭环，在此基础上逐步补齐核心模块，最后通过多轮测试进行数值和性能调优。技术上采用配置驱动组织数据，采用策略模式管理公式，采用组件化方式组织运行逻辑。该路线能够降低耦合并提升后续维护效率。

1.5 论文结构安排
全文共七章。第一章阐述背景、现状、目标和方法。第二章进行需求分析并给出总体架构。第三章详细说明核心系统实现。第四章介绍实验设计与结果分析。第五章讨论性能优化。第六章呈现系统测试与评估。第七章总结全文并提出后续工作方向。

第2章 需求分析与总体架构设计

2.1 需求分析
本项目属于系统驱动型游戏，核心任务是保证循环闭合和节奏稳定。功能上需要覆盖开始场景的角色与模式选择、战斗场景的波次推进、经验与金币成长、商店与升级决策、胜败结算与无尽分支。非功能上需要满足高同屏压力下的性能稳定、模块职责清晰的可维护性、面向配置和策略的可扩展性，以及关键流程可复现可验证的可测试性。需求分析采用需求到模块再到指标的映射方式，确保后续实现可以被验证和追踪。

2.2 核心玩法循环建模
系统核心循环由战斗、结算、升级或商店决策、下一波战斗构成。战斗阶段负责资源产出，决策阶段负责资源转换。生命值、经验值、金币、武器槽位和道具效果是循环中的关键变量。击杀行为带来资源增长，资源增长触发成长选择，成长选择改变属性结构，属性变化再次影响下一波战斗效率。通过这一反馈关系，系统实现可复玩的动态闭环。

2.3 总体架构设计
架构设计采用分层思想。表现层承担界面与反馈，业务层承担流程执行，数据层承担静态配置，策略层承担公式计算，运行支撑层承担会话、上下文和对象复用。`GameMgr`作为流程主控负责状态切换，`GameContext`负责依赖收口，`WaveManager`和`SpawnManager`负责战斗节奏与生成执行，玩家模块负责成长和战斗行为。该结构强调状态集中与能力分散，有助于降低跨模块耦合。

2.4 关键流程设计
开局流程从会话读取开始，完成全局配置加载、玩家实例化和依赖注册后进入运行状态。单波流程围绕波配置驱动刷怪协程，计时结束后执行停刷和清场。波间流程先处理升级选择，再进入商店决策。终局流程根据普通模式或无尽模式分支进行对应结算。整体流程通过状态机统一约束，保证阶段切换一致且可追踪。

2.5 数据驱动与配置规范
项目将模式、波次、敌人、武器、道具、升级需求等规则放入ScriptableObject配置，通过统一校验保证字段完整性与数值合理性。难度倍率、刷怪倍率、商品价格、刷新成本、商品抽取和购买判定等公式由策略资产独立管理。业务代码只负责流程调用，不直接持有复杂公式，从而降低回归风险并提升调参效率。

2.6 本章小结
本章明确了项目的需求边界、循环模型、分层架构和关键流程，建立了从目标到实现的统一约束，为后续系统实现与实验验证提供基础。

第3章 核心系统设计与实现

3.1 会话与状态管理
`GameSessionManager`负责跨场景会话保存，确保开始场景选择结果能够稳定传递到战斗场景。`GameMgr`负责游戏状态管理和时间尺度同步，统一控制等待、进行、暂停、胜利和失败状态。`GameContext`负责注册并提供局内关键服务，减少分散查找和隐式依赖。三者协作形成稳定的流程骨架，保证模块在正确时机获取正确依赖。

3.2 波次与刷怪系统
`WaveConfig`定义波次时长、生成点数量、普通怪权重、特殊怪延迟与强度曲线。`WaveManager`负责波次调度、计时控制、波间流程衔接和无尽扩展。`SpawnManager`负责敌人预加载、生成点采样、预警表现和实例生成。为避免旧协程污染新波次，系统使用会话标识隔离机制，确保越界协程自动失效并提升流程稳定性。

3.3 敌人系统与轻量行为实现
`Enemy`抽象基类统一处理生命、受伤、死亡、掉落、倍率应用和池化重置。`EnemyMelee`实现追击与近战攻击时序，`EnemyHarpoonFish`实现远程投掷行为。系统采用分离力计算减少敌人重叠，使围攻形态更自然且可读。该实现保持了行为复杂度与运行成本之间的平衡，适合当前项目规模。

3.4 玩家成长与资源系统
`PlayerMovement`负责移动输入与位移控制，`StatHandler`负责属性统一计算与上下限钳制，`ExperienceManager`与`LevelSystem`负责经验累计和升级触发，`CoinManager`负责金币收支与事件广播。资源系统与成长系统通过事件串联，确保击杀收益、升级反馈和商店消费形成一致闭环。

3.5 武器与道具系统
`WeaponManager`负责槽位管理、武器添加与移除、合成与重排，`WeaponController`负责自动瞄准、冷却控制和攻击执行。近战命中与远程投射物在表现层与结算层保持一致。道具系统采用`ItemConfig`与`EffectSO`分离设计，`EffectManager`负责效果叠加和移除生命周期，保证多层效果在添加和撤销时都可预测可回退。

3.6 商店与升级决策系统
`UpgradePanel`提供升级选项，`StorePanel`提供刷新、锁定、购买、出售和继续流程，`PostWavesChoicePanel`承接关键波间决策。购买判定通过策略统一返回结果与原因，避免规则散落在界面逻辑中。波间流程采用先升级后商店的顺序，使玩家在做消费决策时拥有更明确的属性信息和收益判断。

3.7 难度与经济策略系统
`DifficultySystem`根据模式与波次上下文统一输出敌人和经济倍率。ValueSystem中的各类策略分别负责敌人缩放、刷怪倍率、商品价格、刷新成本、商品抽取和购买规则。策略化的核心价值在于将公式从业务代码中剥离，使调参更集中、回滚更安全、验证更直接。

3.8 本章小结
本章完成了核心系统的结构与实现说明。各模块在职责边界清晰的前提下形成协作闭环，既满足当前可玩需求，也保留了后续扩展空间。

第4章 数值设计、实验方案与结果分析

4.1 实验目标与评价指标
实验目标包括难度合理性、成长连续性、经济可决策性和性能可承载性。核心指标包括平均生存时长、通关率、无尽轮次、升级频率、金币结余、刷新次数、高阶商品购买率、单位时间击杀和受击频率。通过统一指标体系，能够将体感问题转化为可比较的数据问题。

4.2 实验环境与采样方法
实验在统一版本与场景条件下进行，采用日志记录、手工复盘和Profiler采样结合的方式采集数据。每组参数进行多轮测试并剔除异常样本后统计均值与波动区间，确保结论具有可重复性。对于后期样本不足的问题，补充定向无尽测试以提高数据覆盖度。

4.3 参数组设计
参数调整分为波次节奏、敌人倍率、商店经济和升级曲线四组。波次节奏关注生成间隔和场上上限，敌人倍率关注生命和伤害增长，商店经济关注价格与刷新成本，升级曲线关注等级推进速度。通过分组控制变量，能够准确识别单项参数对整体体验的影响。

4.4 结果分析
实验结果显示，当前参数下难度曲线呈阶段性递进，前期用于学习和建立认知，中期强化挑战，后期强调构筑质量。成长反馈保持连续，刷新行为具有机会成本，商品购买分布随波次推进呈合理变化。对前期过压与中期滚雪球等异常局进行针对性微调后，极端结果比例明显下降，整体稳定性提升。

4.5 本章小结
本章通过分组实验和数据对照验证了参数体系的有效性，证明策略化调参方法能够支持可解释、可追踪和可复验的平衡迭代。

第5章 性能优化与工程化验证

5.1 性能问题定位
项目早期主要瓶颈表现为频繁实例化引发的垃圾回收抖动、波次切换后残留对象干扰、首次资源加载引发卡顿。通过Profiler定位脚本热点和分配峰值后，优化工作围绕对象生命周期和资源加载路径展开。

5.2 对象池优化实现
`GameObjectPoolManager`实现按预制体分类缓存、借出追踪和统一回收，减少重复创建与销毁。波次结束后进行统一回收，保证下一波在干净状态下启动。该机制显著降低了分配波动并平滑帧时间曲线。

5.3 资源加载与生命周期治理
`SpawnManager`在开局阶段完成关键敌人资源预加载，运行期优先命中缓存。界面资源采用按需加载和释放策略。协程与事件订阅在对象禁用或销毁阶段统一清理，避免悬挂任务和重复回调引发隐性性能问题。

5.4 优化效果评估
优化后在高压场景下平均帧率提高，低帧占比下降，垃圾回收峰值减少，波次切换更稳定。尽管仍有进一步优化空间，但当前结果已满足原型阶段的稳定运行与持续迭代需求。

5.5 本章小结
性能优化并非单点技巧，而是流程管理、对象生命周期和资源策略协同的结果。本章验证了工程化优化路径在本项目中的有效性。

第6章 系统测试与结果评估

6.1 测试策略
测试采用功能验证、边界验证、异常验证和压力验证并行推进。每类测试均包含前置条件、操作步骤、预期结果和实际结果，保证测试过程可复现、可审查和可追踪。

6.2 功能测试结果
功能测试覆盖初始化、波次推进、升级选择、商店操作、武器合成与出售、结算与无尽分支等关键路径。结果显示主流程运行稳定，关键交互符合设计预期，系统能够持续完成从开局到终局的完整闭环。

6.3 边界与异常测试结果
边界与异常测试覆盖空配置、余额不足、快速连续操作、依赖晚到、加载失败等场景。通过状态守卫、空值检查、超时等待和降级分支，系统在异常情况下保持可控，不会因单点错误导致整体流程崩溃。

6.4 压力测试结果
在提升同屏敌人数和投射频率的压力场景下，系统保持可玩性，主要瓶颈集中在物理与碰撞开销。该结果与性能分析结论一致，说明后续优化应继续聚焦碰撞层管理、更新节流和高频对象行为开销。

6.5 本章小结
测试结果验证了系统在当前规模下的可靠性与工程可维护性，为论文结论提供了可量化支撑。

第7章 总结与展望

7.1 工作总结
本文围绕Unity3D下Roguelike类PC游戏开发，完成了从需求分析、架构设计、核心实现到参数调优和性能优化的完整实践。项目已经形成可玩、可维护、可扩展的基础框架，并通过测试验证了方案可行性。

7.2 主要贡献
本文的主要贡献在于提出并落地了配置驱动与策略解耦结合的实现路径，构建了战斗、成长、经济和结算的统一闭环，在高同屏场景下通过对象池和生命周期治理提升了系统稳定性，同时建立了指标化实验方法支撑平衡调优。

后续将围绕四个方向推进。第一，扩展敌人行为模型与协同机制。第二，补充地图事件与环境交互。第三，建立自动化数据采集和调参工具。第四，构建跨局成长与解锁系统，提升长期目标感与复玩深度。

摘  要
随着独立游戏市场的持续发展，融合Roguelike机制与实时生存玩法的“类幸存者”游戏逐步成为动作游戏的重要方向。该类型通常具备战斗节奏快、随机成长路径多、重玩价值高等特征，同时对系统架构、数值平衡和运行稳定性提出更高要求。本文以本人基于Unity3D开发的2D俯视角PC游戏原型为研究对象，围绕核心玩法闭环、模块化系统实现、数值调优与性能优化开展研究与实践。

在系统设计上，本文以“波次战斗-资源结算-升级/商店决策-下一波战斗”为主循环，构建了由会话与状态管理、波次与刷怪系统、敌人轻量AI、玩家成长与武器道具系统、商店升级系统、难度与经济策略系统、对象池与资源管理组成的完整技术链。在工程实现上，项目采用ScriptableObject配置驱动与策略模式解耦，将核心平衡公式集中到ValueSystem策略资产中，实现“流程代码与平衡公式分离”。在运行优化上，通过对象池复用、预加载和生命周期治理降低运行时分配抖动，提升高负载场景稳定性。

在验证层面，本文基于多轮游玩测试与参数对照实验，从生存时长、升级节奏、金币流动、购买分布、无尽轮次与性能指标等维度评估系统表现。结果表明：该原型已形成完整且可复玩的玩法闭环；通过策略化调参，难度推进与经济节奏具备较好可解释性；通过工程优化，系统在高同屏压力下运行稳定性明显改善。

ABSTRACT
This thesis presents the design and implementation of a Unity3D-based Roguelike PC game prototype with survivor-like real-time combat. The project focuses on building a complete gameplay loop, maintainable engineering architecture, tunable balancing strategies, and stable runtime performance under high enemy density.

The core loop is defined as wave combat, resource settlement, upgrade/store decision, and next-wave progression. The implemented system includes session and state management, wave scheduling and spawning, lightweight enemy AI, player growth and weapon/item build systems, store and upgrade flow, strategy-based difficulty/economy balancing, and pooling-based runtime optimization.

Evaluation is conducted through iterative playtests and parameter-group experiments, with metrics such as survival duration, level-up rhythm, coin economy, purchase distribution, endless progression, and profiler-based performance indicators. Results indicate that the prototype achieves a complete and replayable gameplay loop and improved runtime stability after optimization.

Keywords: Unity3D, Roguelike, Wave System, Data-Driven Design, Strategy Pattern, Balance, Object Pooling

第1章 绪论
1.1 研究背景与意义
Roguelike与Roguelite玩法在独立游戏中长期保持活跃，类幸存者分支通过“低操作门槛+高策略密度+短局高反馈”形成稳定受众。与传统线性关卡不同，该类型更依赖系统循环质量：玩家在战斗中获取资源，在波间进行成长决策，并在下一波验证构筑强度。若循环任一环节失衡，体验会迅速恶化。
[图1-1 插图位置] Roguelike类幸存者核心体验模型图（战斗压力-成长反馈-随机决策）。

1.2 国内外研究现状
国外实践强调“操作简化而决策深化”，并普遍采用轻量AI与对象池优化应对高同屏压力；国内在课程项目与社区实践中已积累大量模块级经验，但“玩法闭环+工程架构+实验验证”一体化案例相对不足。本文在此基础上给出完整实现路径与验证过程。
[表1-1 插图位置] 国内外研究对比表（机制、架构、性能验证、可复用性）。

1.3 研究目标与研究内容
本文目标是基于Unity3D构建一款具备完整核心循环的Roguelike类PC游戏原型，并形成可扩展、可调参与可优化的工程框架。核心内容包括会话与状态管理、波次与刷怪系统、敌人行为系统、玩家成长与构筑系统、商店升级决策、难度与经济策略、对象池优化与测试评估。
[表1-2 插图位置] 研究目标-模块-验证指标映射表。

1.4 研究方法与技术路线
采用“迭代开发+数据复盘”的路线：需求分析 -> 架构设计 -> 核心实现 -> 数值与性能调优 -> 测试评估 -> 结论总结。方法上采用配置驱动、策略解耦、组件化协作和实验驱动调参。
[图1-2 插图位置] 技术路线图。

1.5 论文结构安排
第2章给出需求与总体架构；第3章阐述核心系统实现；第4章进行数值实验与结果分析；第5章讨论性能优化；第6章给出测试评估；第7章总结与展望。

第2章 需求分析与总体架构设计
2.1 需求分析
2.1.1 功能需求
（1）BeginScene选择流程：玩家在开始场景选择角色、武器和模式，并写入`GameSessionManager`。
（2）GameScene战斗流程：支持移动、自动攻击、敌人生成、波次推进、升级与商店决策。
（3）成长构筑流程：支持经验升级、武器购买/出售/合成、道具效果叠加和金币循环。
（4）终局分支流程：支持普通胜败结算与无尽衔接。
2.1.2 非功能需求
（1）性能稳定性；（2）可维护性；（3）可扩展性；（4）可测试性。
[表2-1 插图位置] 功能需求与模块映射表。
[表2-2 插图位置] 非功能需求验证表。

2.2 核心玩法循环建模
系统主循环为“战斗波次 -> 结算转换 -> 升级/商店决策 -> 下一波战斗”。关键变量为生命、经验、金币、武器槽和效果层数。击杀、升级、购买和属性变化通过事件链互相反馈，形成可复玩的动态闭环。
[图2-1 插图位置] 核心循环图。
[图2-2 插图位置] 游戏状态机图（Waiting/Playing/Paused/Victory/Defeat）。

2.3 总体架构设计
系统分为表现层、业务层、数据层、策略层、运行支撑层。`GameMgr`负责状态主控，`GameContext`负责依赖注册与查询，`WaveManager`与`SpawnManager`负责战斗调度，ValueSystem负责平衡公式。
[图2-3 插图位置] 初始化时序图（GameSession -> GameMgr -> GameContext -> WaveManager）。

2.4 关键流程设计
开局：读取会话、加载配置、实例化玩家、注册上下文。
单波：读取`WaveConfig`、刷怪协程运行、计时结束后停刷清场。
波间：升级面板后进入商店决策。
终局：普通结算或无尽分支。
[图2-4 插图位置] 波次生命周期活动图。

2.5 数据驱动与配置规范
核心配置为`ModeConfig`、`WaveConfig`、`EnemyConfig`、`WeaponConfig`、`ItemConfig`、`LevelConfig`；核心策略为敌人缩放、刷怪倍率、商品价格、刷新价格、商品抽取与购买规则。遵循“配置定义内容，策略定义公式，业务执行流程”。
[表2-3 插图位置] 关键配置项说明表。

2.6 本章小结
本章明确了需求边界、循环模型、分层架构与关键流程，为第3章实现提供统一约束。

第3章 核心系统设计与实现
3.1 会话与状态管理
`GameSessionManager`负责跨场景会话保持；`GameMgr`负责状态切换与时间尺度同步；`GameContext`负责依赖注册与服务定位，保证模块协作边界清晰。

3.2 波次与刷怪系统
`WaveConfig`以普通怪权重池和特殊怪延迟池组织波次数据；`WaveManager`负责波次调度、波间流程与无尽衔接；`SpawnManager`负责预加载、生成点采样和预警落怪。`spawnSessionId`用于防止跨波协程残留。

第5章 性能优化与工程化验证
5.1 性能问题定位
主要问题为频繁实例化导致GC抖动、波次切换残留对象与首次加载卡顿。
5.2 对象池优化
`GameObjectPoolManager`实现缓存、借出追踪与统一回收，降低实例化开销。
5.3 资源加载与生命周期管理
`SpawnManager`预加载敌人资源；UI按需加载；协程与事件在禁用/销毁时统一清理。
5.4 优化效果评估
优化后平均帧率上升、低帧占比下降、GC峰值降低，流程稳定性提升。
