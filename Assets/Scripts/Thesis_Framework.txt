摘  要

随着独立游戏市场的持续发展，融合Roguelike机制与实时生存玩法的“类幸存者”游戏逐步成为动作游戏的重要方向。该类型通常具备战斗节奏快、随机成长路径多、重玩价值高等特征，同时对系统架构、数值平衡和运行稳定性提出更高要求。本文以本人基于Unity3D开发的2D俯视角PC游戏原型为研究对象，围绕核心玩法闭环、模块化系统实现、数值调优与性能优化开展研究与实践。

在系统设计上，本文以“波次战斗-资源结算-升级/商店决策-下一波战斗”为主循环，构建了由会话与状态管理、波次与刷怪系统、敌人轻量AI、玩家成长与武器道具系统、商店升级系统、难度与经济策略系统、对象池与资源管理组成的完整技术链。在工程实现上，项目采用ScriptableObject配置驱动与策略模式解耦，将核心平衡公式集中到ValueSystem策略资产中，实现“流程代码与平衡公式分离”。在运行优化上，通过对象池复用、预加载和生命周期治理降低运行时分配抖动，提升高负载场景稳定性。

在验证层面，本文基于多轮游玩测试与参数对照实验，从生存时长、升级节奏、金币流动、购买分布、无尽轮次与性能指标等维度评估系统表现。结果表明：该原型已形成完整且可复玩的玩法闭环；通过策略化调参，难度推进与经济节奏具备较好可解释性；通过工程优化，系统在高同屏压力下运行稳定性明显改善。

关键词：Unity3D；Roguelike；波次系统；配置驱动；策略模式；数值平衡；对象池优化

ABSTRACT

This thesis presents the design and implementation of a Unity3D-based Roguelike PC game prototype with survivor-like real-time combat. The project focuses on building a complete gameplay loop, maintainable engineering architecture, tunable balancing strategies, and stable runtime performance under high enemy density.

The core loop is defined as wave combat, resource settlement, upgrade/store decision, and next-wave progression. The implemented system includes session and state management, wave scheduling and spawning, lightweight enemy AI, player growth and weapon/item build systems, store and upgrade flow, strategy-based difficulty/economy balancing, and pooling-based runtime optimization.

Evaluation is conducted through iterative playtests and parameter-group experiments, with metrics such as survival duration, level-up rhythm, coin economy, purchase distribution, endless progression, and profiler-based performance indicators. Results indicate that the prototype achieves a complete and replayable gameplay loop and improved runtime stability after optimization.

Keywords: Unity3D, Roguelike, Wave System, Data-Driven Design, Strategy Pattern, Balance, Object Pooling

第1章 绪论

1.1 研究背景与意义
Roguelike与Roguelite玩法在独立游戏中长期保持活跃，类幸存者分支通过“低操作门槛+高策略密度+短局高反馈”形成稳定受众。与传统线性关卡不同，该类型更依赖系统循环质量：玩家在战斗中获取资源，在波间进行成长决策，并在下一波验证构筑强度。若循环任一环节失衡，体验会迅速恶化。

ABSTRACT
========================

This thesis presents the design and implementation of a Unity3D-based Roguelike PC game prototype with survivor-like real-time combat. The project focuses on building a complete gameplay loop, maintainable engineering architecture, tunable balancing strategies, and stable runtime performance under high enemy density.

The core loop is defined as wave combat, resource settlement, upgrade/store decision, and next-wave progression. The implemented system includes session and state management, wave scheduling and spawning, lightweight enemy AI, player growth and weapon/item build systems, store and upgrade flow, strategy-based difficulty/economy balancing, and pooling-based runtime optimization. A data-driven approach using ScriptableObject and strategy assets is adopted to decouple gameplay flow from balancing formulas.

Evaluation is conducted through iterative playtests and parameter-group experiments, with metrics such as survival duration, level-up rhythm, coin economy, purchase distribution, endless progression, and profiler-based performance indicators. Results indicate that the prototype achieves a complete and replayable gameplay loop, supports explainable balancing iteration, and shows improved runtime stability after optimization.

This work provides a practical engineering reference for small-scale Roguelike survivor development and establishes a foundation for future expansion in enemy behavior depth, map event design, and balancing automation.

Keywords: Unity3D, Roguelike, Wave System, Data-Driven Design, Strategy Pattern, Balance, Object Pooling

1.1 研究背景与意义

摘  要

随着独立游戏市场的持续发展，融合Roguelike机制与生存玩法的“类幸存者”游戏逐渐成为动作游戏的重要类型。该类游戏通常具有战斗节奏快、成长路径随机、重玩价值高等特点，同时对系统组织、数值平衡与运行稳定性提出更高要求。
本研究以一款2D俯视角PC游戏原型为对象，围绕核心玩法循环构建、模块化系统设计、参数调优与性能验证展开。研究在需求分析基础上，形成了“战斗推进—资源结算—成长选择—阶段决策”的整体流程，并据此完成相关功能模块的实现与联动。
在多轮游玩测试与迭代调整中，研究重点考察了难度曲线、经济节奏和运行性能等关键指标。结果表明，该原型能够形成较完整且可复玩的玩法闭环；通过持续调优，系统在成长反馈、挑战递进和策略选择方面表现出较好的一致性与可解释性；在高负载场景下，运行稳定性也得到明显改善。
本研究说明，在中小规模开发条件下，采用配置化、模块化与迭代优化相结合的方法，能够有效支撑Roguelike类生存游戏的设计与实现。相关实践为后续扩展玩法深度、完善平衡方法与提升工程效率提供了参考基础。

第1章 绪论

1.1 研究背景与意义

近年来，Roguelike与Roguelite玩法在独立游戏与中小型商业项目中持续受到关注。特别是“类幸存者”分支通过简化操作负担、提高同屏压力和强化成长反馈，形成了明显的市场辨识度。与传统关卡推进型动作游戏不同，该类游戏并不依赖复杂剧情或高成本关卡制作，而是依赖系统循环设计：玩家在每一局中通过战斗获得资源，再通过升级和商店决策将资源转化为战斗能力，并在下一波中验证构筑效果。这一循环的质量直接决定了玩家留存与复玩意愿。

对我而言，本课题的价值不仅是完成“一个能运行的Demo”，更是探索“个人规模项目如何在有限开发资源下实现系统闭环、可维护架构与可调平衡”。在实际开发过程中，我发现类幸存者游戏最容易出现三类问题：第一，系统功能看似齐全但循环断裂，玩家无法形成持续目标；第二，数值配置散乱且公式耦合在业务代码中，后续调参成本极高；第三，大量敌人与投射物造成明显掉帧，体验不稳定。为解决以上问题，我将研究重点放在架构与实现并重：既保证功能落地，也保证后续迭代效率。

本课题的实践意义主要体现在以下方面。其一，工程方法意义。通过配置驱动与策略解耦，我将平衡逻辑从业务流程中分离，降低维护风险。其二，系统设计意义。通过状态管理、波次管理、商店升级、敌人行为和对象池优化形成完整技术链。其三，教学与能力训练意义。项目将软件工程、实时交互、数值设计、性能分析等知识统一到可验证的开发流程中，为后续从事游戏开发工作提供实践基础。

1.2 国内外研究现状

国外在Roguelike相关研究与实践方面起步较早，相关成果不局限于论文，还包括开发者大会演讲、开源项目与技术博客。总体经验表明：优秀作品通常保持“操作简单但决策复杂”，并通过短周期反馈让玩家迅速感知成长差异。在技术实现方面，面对高同屏敌人压力，实践者倾向采用轻量行为模型与池化优化，而非一味追求复杂AI。

国内近年来在课程实践与独立开发社区中同样积累了大量经验。许多项目已覆盖地牢生成、道具系统、敌人行为和基础数值平衡等模块。但从工程闭环角度看，仍存在“功能分散、架构弱、测试与性能验证不足”的问题。特别是在个人项目中，若缺乏统一的状态与配置管理，后期改动极易引发连锁回归。

综合现状可以看出，现有研究中仍缺少兼顾“玩法闭环、工程可维护、数值可解释、性能可验证”的完整案例。本文尝试通过真实项目代码与迭代数据，给出一套可复用的实现路径。

1.3 研究目标与研究内容

本人本课题的总体目标是：在Unity3D环境下完成一款具备完整核心循环的Roguelike类PC游戏原型，并形成可扩展、可调参与可优化的工程化框架。围绕该目标，我将任务拆分为以下六项：

（1）实现稳定的局内状态与会话管理，确保从开局到结算流程可追踪。

（2）构建可配置波次系统，支持普通怪与特殊怪混合生成、波内强度变化及无尽衔接。

（5）在高同屏场景下实施对象池与资源管理优化，提高运行稳定性。

第2章 需求分析与总体架构设计

2.1 需求分析

2.1.1 功能需求

（1）玩家控制与战斗需求。玩家需持续移动规避并执行自动攻击，支持近战与远程两类攻击形态。

（2）波次推进需求。系统按波次持续生成敌人，支持不同敌人组合与节奏变化，并可在波间切换到决策阶段。

（3）成长构筑需求。玩家通过击杀获得经验和金币，触发升级选择与商店购买，形成差异化构筑。

（4）经济循环需求。需支持购买、刷新、锁定、出售、合成等行为，并保证资源扣减与收益回流一致。

（5）流程控制需求。系统需覆盖开局、战斗、波间、胜败结算与无尽分支，状态切换无冲突。

2.1.2 非功能需求

（1）性能稳定性：高同屏下可持续游玩，避免频繁卡顿。

（2）可维护性：模块职责清晰，减少脚本间隐式耦合。

（3）可扩展性：新增内容优先通过配置和策略完成。

（4）可测试性：关键路径可复现、可记录、可判定。

2.2 核心玩法循环建模

我将本项目建模为“战斗波次 -> 资源结算 -> 升级/商店决策 -> 下一波战斗”的循环系统。战斗阶段负责资源产出，波间阶段负责资源转换，二者交替构成完整体验。该模型的关键变量为生命、经验、金币、武器槽位和道具效果。变量之间通过事件链相互作用：击杀触发经验与掉落，升级触发面板，面板选择影响属性，属性反过来影响下一波生存效率。

在节奏层面，我采用“波内升压 + 波间喘息”的双相结构。波内由刷怪强度曲线驱动，跨波由模式倍率与无尽增长驱动。该设计既保证前期学习空间，也保证后期挑战上限。

2.3 总体架构设计

2.3.1 分层架构

系统分为五层：表现层、业务层、数据层、策略层、运行支撑层。表现层负责UI与反馈；业务层负责流程执行；数据层负责配置资产；策略层负责平衡公式；运行层负责上下文与池化管理。该分层可减少跨层耦合，便于定位问题与独立迭代。

2.3.2 关键对象关系

`GameMgr`负责状态主控与初始化；`GameContext`负责服务注册与依赖查询；`WaveManager`负责波次节奏；`SpawnManager`负责生成执行；玩家组件负责成长与战斗；UI面板负责波间决策。对象关系遵循“状态集中、能力分散”的原则：集中管理状态，分散实现能力。

第3章 核心系统设计与实现

3.1 游戏会话与状态管理系统

3.1.1 GameSessionManager跨场景会话管理

在项目早期，我将会话信息放在多个场景脚本中临时保存，结果在场景切换后经常出现引用丢失与状态不一致。为解决该问题，我实现`GameSessionManager`单例并使用`DontDestroyOnLoad`保持跨场景有效。该模块在启动时加载`GameSession`，并通过`GetSession()`和`GetSessionAsync()`提供访问接口。会话中包含选中角色、武器和模式等关键输入，是整个局内系统的起点。

我特别强调会话读写边界：场景准备阶段可写入，局内运行阶段原则上只读。这样可避免中途被其他脚本改写导致难以追踪的问题。事实证明，明确边界后，初始化链路的稳定性明显提升。

3.1.2 GameMgr状态控制与时间尺度同步

`GameMgr`是局流程主控。它负责地址资源初始化、全局配置加载、玩家实例化、组件补全、状态切换和终局展示。状态枚举包含`Waiting`、`Playing`、`Paused`、`Victory`、`Defeat`。我将时间尺度同步封装在状态切换中，确保暂停与结算阶段的逻辑冻结一致。

在实现上，`GameMgr`通过静态事件广播状态变化，其他模块按需订阅。该模式减少了直接耦合，但也要求严格的订阅解绑管理。为避免悬挂引用，我在`OnDisable/OnDestroy`阶段统一取消订阅，确保生命周期可控。

我还引入`GameEndPresentationKind`用于区分“逻辑状态”和“展示类型”。该设计解决了无尽模式死亡时“逻辑失败但展示为成绩结算”的冲突问题，使UI层判断更清晰。

3.1.3 GameContext依赖注册与服务定位

`GameContext`作为局内依赖中心，集中登记`GameMgr`、`WaveManager`、`SpawnManager`、玩家对象及其关键组件。业务脚本通过`TryGetXxx`接口获取依赖，替代全局查找。该方案的核心收益是提高可维护性与容错性：依赖未就绪时可优雅等待或降级，排查问题时也可快速定位注册状态。

3.2 波次与刷怪系统

3.2.1 WaveConfig结构设计

`WaveConfig`以波次列表为核心，每个`WaveData`描述时长、启动延迟、普通怪池、特殊怪池、生成点数量、刷怪节奏、同屏上限和强度曲线。普通怪使用加权抽取；特殊怪使用定量延迟生成。该结构既可表达稳定压力，也可表达阶段爆点。

我在参数设计中加入了多重约束，例如最小间隔、最小批量、上限限制等，目的是在配置层预防失控参数进入运行时。通过这些约束，调参时的风险显著降低。

3.2.2 WaveManager调度实现

`WaveManager`负责波次推进、计时、刷怪协程、波间升级与商店流程，以及普通结局与无尽分支。为处理复杂时序，我引入`spawnSessionId`机制，使旧协程在波次切换后自动失效，避免越波生成。

战斗结束时，我统一执行“停刷 -> 清场 -> 隐藏战斗面板 -> 升级流程 -> 商店流程”。该顺序稳定且可复现，有利于测试与调试。

3.2.3 SpawnManager生成执行

`SpawnManager`负责预加载敌人资源、生成点采样、预警显示与对象池生成。预加载会遍历全波次敌人配置并去重加载，运行时直接命中缓存。生成点由地图范围随机采样，每波可配置数量。普通怪在锚点周围簇生，特殊怪按延迟独立投放。

在预警生成中，我通过协程与中止回调保证波次结束时可立即取消落怪并回收预警对象，避免流程残留。

3.2.4 无尽模式衔接

无尽模式采用“复用最后一波模板 + 难度指数增长”的方式。该方案开发成本低、维护压力小，同时保持挑战持续提升。通过`DifficultySystem`更新当前波状态后，敌人属性与刷怪倍率会同步放大。

3.3 敌人系统与轻量AI实现

3.3.1 Enemy抽象基类

`Enemy`基类统一处理生命、受伤、死亡、掉落、难度倍率应用、玩家查找、边界约束与分离力计算。`InitializeForSpawn`用于对象池复用时重置状态。该设计使不同敌人在结算层保持一致，降低重复实现成本。

分离力实现是本项目轻量AI的重要组成。通过邻域检测计算排斥向量，敌人可在接近玩家时形成松散围攻而非重叠堆叠，提升观感与可读性。

3.3.2 EnemyMelee近战行为

`EnemyMelee`实现“追击 -> 进入攻击圈 -> 动画命中帧结算 -> 动画结束解锁”的流程。移动速度由追踪力和分离力合成，攻击通过动画事件触发。该机制保证视觉与结算一致，避免“动作未到位先掉血”的违和。

近战击退采用持续时间插值，期间暂停追击。该处理增强打击反馈并给玩家提供可感知的操作收益。

3.3.3 EnemyHarpoonFish远程行为

鱼叉敌人实现双圈层行为：兴奋圈外随机游走，兴奋圈内追击，投掷圈内停下并投射。投掷方向在起手时锁定，命中释放事件只负责按锁定方向生成投射物。该方案兼顾行为稳定性与表现一致性。

3.3.4 难度倍率统一

敌人最终属性由`DifficultySystem`统一输出倍率，再在基类应用到生命、伤害、速度和攻击冷却。统一入口减少了多脚本各自放大造成的误差。

3.4 玩家成长与战斗系统

3.4.1 移动与属性计算

玩家移动由`PlayerMovement`读取输入并驱动刚体速度。属性系统`StatHandler`采用统一公式计算并在末端夹取上下限，基础值来源于全局配置与角色初始修正，后续增益来自道具效果与其他模块修正。

3.4.2 经验与等级系统

`ExperienceManager`监听经验事件并累计总经验，达到阈值后调用`LevelSystem`升级。`LevelSystem`负责等级状态管理与升级事件触发。两者分离使经验计算与等级状态更清晰，也便于单独测试。

3.4.3 金币系统

`CoinManager`统一管理金币增减与余额查询，并通过事件驱动UI刷新。商店扣费、出售返还、局末清零均通过该模块完成，避免资金状态分散在多个脚本。

3.5 武器系统设计与实现

3.5.1 WeaponManager槽位管理

`WeaponManager`管理7个挂载位与武器槽逻辑。单武器时使用中心挂载位，多武器时分布到环形挂载位。模块支持购买添加、移除出售、手动/快速合成与槽位压紧重排。该设计保证显示与逻辑同步，减少无效空位。

3.5.2 WeaponController攻击执行

`WeaponController`以挂载位为单位维护攻击冷却，自动寻找最近敌人并触发攻击。远程武器生成投射物，近战武器通过命中盒与动画事件结算伤害。伤害计算会综合武器伤害、玩家属性、暴击与修饰器结果。

3.5.3 近战瞄准锁定与命中一致性

为了避免近战挥砍期间朝向抖动，我在攻击开始时冻结挂载位旋转，在动画结束事件解锁。该机制显著减少了“看起来砍中却未命中”的主观落差。

3.5.4 武器合成与经济联动

武器满槽后优先判断是否可合成。同名同型同级可升级为更高等级运行时副本。出售价格通过价格策略统一计算，使买卖规则一致。

3.6 道具与效果系统

3.6.1 容器与逻辑分离

`ItemConfig`作为道具容器，保存展示信息、分类、价格与效果引用；`EffectSO`作为效果基类，定义应用与移除生命周期。容器与逻辑分离可提高效果复用率。

3.6.2 StatEffectSO与LogicEffectSO

`StatEffectSO`用于属性修正，`LogicEffectSO`用于动态组件效果。逻辑效果支持`Single`与`Stackable`堆叠模式，满足不同效果叠加需求。

3.6.3 EffectManager堆叠管理

`EffectManager`记录效果层数并管理逻辑组件注册/反注册。移除逻辑按层执行，最后一层才真正卸载，避免堆叠道具异常失效。对象销毁时统一清理全部效果，防止残留组件。

3.7 商店与升级系统

3.7.1 StorePanel流程

`StorePanel`在波间展示4个商品槽，支持刷新、锁定、购买、出售、合成、继续。刷新时锁定槽优先保留并前置，未锁定槽重新抽取。面板显示时重新绑定玩家与管理器引用，避免跨局引用失效。

3.7.2 购买规则策略化

购买判定由`ShopPurchaseRuleStrategy`返回`ShopPurchaseDecision`，包含可买性、支付价格和失败原因。业务层仅执行流程，不硬编码规则，提升可维护性。

3.7.3 UpgradePanel升级选择

升级面板每次随机提供3个道具，按玩家等级对品阶权重进行偏置。玩家可选择一个道具或跳过。该机制在中后期能稳定提供成长选择，避免长期无反馈。

3.7.4 波间决策与节奏关系

我采用“先升级后商店”的顺序。先升级可让玩家以最新属性评估商店收益，减少盲买。测试中该顺序的决策清晰度更高。

3.8 难度与经济策略系统

3.8.1 DifficultySystem入口

`DifficultySystem`作为难度与经济参数读取中心，按当前波次与无尽状态返回敌人倍率、刷怪倍率、材料掉率等结果。各模块通过接口读取，避免重复实现公式。

3.8.2 商品价格策略

`ShopProductPriceStrategy`基于基础价、模式倍率和波次增长计算最终价格。线性与二次增长组合使前期温和、后期加压，维持经济挑战曲线。

3.8.3 刷新价格策略

`ShopRefreshPriceStrategy`结合当前波次与本店刷新次数计算刷新成本，避免无限低成本刷店破坏经济平衡。

3.8.5 购买判定策略

`ShopPurchaseRuleStrategy`统一处理余额不足、武器槽限制、合成条件、商品合法性等规则。策略化判定让商店流程更稳健。

ABSTRACT

With the rapid growth of the indie game market, real-time survival games that integrate Roguelike mechanics have become an influential direction in action game design. This genre requires a delicate balance among high-pressure combat, randomized build progression, and replayability, while also imposing strict requirements on architecture, balancing, and runtime performance. This thesis focuses on the design and implementation of a Unity3D-based Roguelike PC game, taking my independently developed prototype as the research subject.

The project is built around a complete gameplay loop: wave combat, resource settlement, upgrade/store decision, and next-wave progression. A modular solution is constructed, including game state management, wave scheduling, enemy spawning, player growth, weapon and item systems, store and upgrade systems, difficulty strategies, and object pooling optimization. ScriptableObject-based data-driven design and strategy-oriented balancing are adopted to decouple formulas from business logic. In other words, gameplay scripts execute flow, while balancing formulas are encapsulated in strategy assets.

Core implementations include: stable state transitions using GameMgr and GameContext; configurable wave progression based on WaveConfig and WaveManager; lightweight enemy AI with melee swarm behavior and ranged dual-radius behavior; and a growth ecosystem combining StatHandler, EffectManager, WeaponController, WeaponManager, and StorePanel. The current framework supports iterative expansion and balancing with maintainable engineering structure.

Evaluation is conducted through iterative playtests and parameter tuning, using metrics such as survival duration, level-up rhythm, currency flow, purchase distribution, and endless-round progression. Results indicate that the prototype achieves a complete playable loop and relatively smooth difficulty ramping under a solo development scope. Performance and runtime stability are significantly improved through object pooling and resource lifecycle control.

This work provides a reusable engineering reference for small-scale Roguelike survival development and lays the groundwork for future studies on advanced enemy behavior, map event systems, balancing automation, and long-term meta progression.

Keywords: Unity3D, Roguelike, Wave System, Data-Driven Design, Strategy Pattern, Balance, Object Pooling

第1章 绪论

1.1 研究背景与意义

近年来，Roguelike与Roguelite类游戏在独立游戏市场中保持高热度。其中，以“幸存者”玩法为代表的实时动作生存分支，通过低门槛操作、高密度敌人、随机成长路径和短局循环机制，建立了较强的传播能力与复玩价值。与传统线性关卡游戏相比，此类作品更依赖系统间联动：敌人增长速度、玩家成长速度、经济消耗速度、刷新随机性与反馈节奏必须在有限时长内达成动态平衡。一旦其中某一环节失衡，玩家体验会迅速恶化，例如前期过难导致流失，或后期过强导致无挑战感。

从工程实践角度看，开发一款Roguelike类生存游戏并非只是“把若干功能拼起来”。在真实项目中，我需要同时处理多个层面的难题：其一，玩法闭环是否完整；其二，核心系统是否可维护、可扩展；其三，大量单位同屏时是否仍能稳定运行。特别是在个人独立开发场景下，时间与人力资源有限，架构选择错误会直接导致后期维护成本指数增长。基于以上原因，我将本课题聚焦为“在Unity3D中构建一个可玩、可调、可扩展、可优化的Roguelike类PC游戏框架”，并通过完整实现验证方案可行性。

本研究的意义主要体现在三个方面。第一，方法意义：我尝试将“配置驱动 + 策略模式 + 组件化系统 + 性能优化”组合成可复用技术路径，为同类小型项目提供实施范式。第二，工程意义：在保证可玩性的前提下，建立了清晰的系统职责边界与参数管理机制，降低了后续迭代风险。第三，教学与实践意义：本项目将软件工程、实时交互设计、数值设计、性能分析等知识整合到同一开发流程中，强化了“从设计到落地再到验证”的完整能力链。

国外在Roguelike/Roguelite设计领域积累较早，相关资料不仅包括学术论文，也包括大量开发者会议演讲、技术博客与后验复盘文章。总体上，国外实践强调两点：第一，操作复杂度应低于策略复杂度，即玩家操作可以简单，但决策必须有深度；第二，系统反馈要“短周期、强可见”，确保玩家能在数分钟内感知成长路径差异。在敌人实现上，许多高同屏项目倾向采用轻量行为而非复杂寻路，以换取可控性能和稳定压迫感。

国内的研究与实践近年发展迅速。高校课程与毕业设计中，Roguelike相关选题数量明显增加，内容覆盖关卡随机、数值平衡、装备系统等方向；开发社区中也有大量Unity实战经验分享。然而，现有成果多集中于单点功能实现，完整覆盖“玩法闭环 + 工程架构 + 平衡策略 + 性能验证”的系统化案例相对不足。尤其在“个人规模项目如何保证后期可维护性”这一问题上，可复用经验仍不充分。

1.4 研究方法与技术路线

第三，组件化协作方法。基于Unity组件模型组织玩家、敌人、UI和管理器脚本，强调单一职责与显式依赖。

第四，实验驱动调优方法。围绕生存时长、升级节奏、资源流入流出等指标进行多轮实测与对比，避免凭经验盲调。

2.1 需求分析

本项目不是关卡脚本驱动型短流程游戏，而是可重复运行的系统驱动型游戏。因此在需求定义时，我优先关注“系统循环是否闭合、参数是否可控、工程是否可维护、运行是否可稳定”，并将需求划分为功能需求与非功能需求两大类。

2.1.1 功能需求

（1）玩家移动与攻击功能。玩家需在二维平面连续移动并自动攻击敌人，系统应支持近战与远程武器类型，同时在命中、伤害、暴击、吸血等反馈上保持一致的结算逻辑。该需求在项目中由`PlayerMovement`、`WeaponController`、`WeaponManager`、`StatHandler`等模块协同完成。

（2）波次战斗功能。系统需按波次推进战斗，支持普通怪持续生成与特殊怪事件生成，且可配置波时长、刷怪强度、同屏上限与预警展示。该需求由`WaveConfig`、`WaveManager`、`SpawnManager`构成主干实现。

（3）敌人行为功能。敌人需具备可识别的行为模式并产生有效威胁。当前项目实现了近战追击型与投掷型两类敌人，支持攻击冷却、受击、死亡、掉落、击退、地图边界约束等能力，满足原型阶段的行为多样性需求。

（4）成长构筑功能。玩家在战斗中获得经验并触发升级选择，在波间通过商店购买道具或武器，以形成差异化构筑。该需求涉及`ExperienceManager`、`LevelSystem`、`UpgradePanel`、`StorePanel`、`InventoryManager`与`EffectManager`。

（5）经济循环功能。玩家应能在局内获取金币、消耗金币、出售武器回收金币，并通过刷新机制进行策略博弈。该功能由`CoinManager`与商店策略系统共同实现，形成可验证的资源闭环。

（6）流程控制功能。系统需完成开局初始化、战斗进行、波间暂停、终局结算、无尽分支等全流程，并保证状态切换不冲突。该需求由`GameMgr`与`WaveManager`共同管理。

2.1.2 非功能需求

（1）实时性能要求。在大量敌人同屏、子弹频繁生成、UI持续刷新的情况下，系统应尽量维持稳定帧率与可接受输入响应。项目采用对象池与预加载方案降低运行时抖动。

（2）可维护性要求。模块职责需明确，避免“超级管理器”脚本。当前设计中状态、刷怪、攻击、商店、数值策略等均由独立模块承担，并通过`GameContext`集中管理依赖引用。

（3）可扩展性要求。新增敌人、武器、道具、难度模式时，不应重写核心流程。项目通过配置驱动和策略资产实现扩展，业务脚本复用率较高。

（4）可测试性要求。关键流程必须可观察、可复现、可定位。例如波次切换、商店购买、升级触发、无尽转换等路径应有明确状态条件和边界处理逻辑。

2 需求分析与总体架构设计（建议 3500~4200字）

2.1 需求分析
本课题面向Roguelike类生存玩法，围绕“短时高压战斗 + 波间决策构筑 + 多轮重复游玩”建立需求模型。与传统线性关卡不同，本项目重点不在固定剧情推进，而在可重复运行的系统闭环。因此，需求分析不仅包括可见功能，还必须覆盖战斗节奏、数值演进、工程可维护性与性能边界。为保证后续实现可验证，本文采用“需求项 -> 代码模块 -> 可观测指标”的三段式描述，具体对应关系见表2-1。

2.1.1 功能需求
（1）角色控制与战斗执行需求。玩家需要在2D平面完成高频移动规避，同时系统持续提供自动或半自动攻击输出能力。该需求在代码层由`PlayerMovement`、`WeaponController`、`WeaponManager`协同实现，其中移动与攻击解耦，保证玩家操作负担可控且战斗反馈连续。

（2）敌人生成与行为需求。系统需要根据波次阶段不断生成敌人，并在可控成本下形成“包围压迫感”。该需求由`WaveManager`调度、`SpawnManager`执行、`Enemy`及其子类表现。普通敌人采用加权随机生成，特殊敌人采用延迟与定量生成，以支持“稳定压力 + 阶段峰值”的节奏设计。

（3）成长与构筑需求。玩家应在战斗过程中通过经验升级、道具叠加、武器扩展持续获得成长反馈。该部分由`ExperienceManager`、`LevelSystem`、`InventoryManager`、`EffectManager`、`StatHandler`构成，分别承担经验累计、等级推进、道具容器、效果生命周期与属性结算职责。

（4）经济与决策需求。波间阶段需提供明确且可比较的资源决策：购买、刷新、锁定、合成、出售。项目以`StorePanel`和`UpgradePanel`承载交互流程，并通过`CoinManager`、`WeaponManager`与ValueStrategy策略系统约束价格、概率和购买规则，避免“随机过强”或“策略无效”。

（5）局流程控制需求。系统需完整覆盖开局、战斗中、波间暂停、终局结算和无尽分支，且状态切换应可追踪、可恢复。该需求由`GameMgr`状态机与`WaveManager`波次逻辑联合实现，保证流程边界明确，便于后续测试。

2.1.2 非功能需求
（1）性能稳定性需求。在高敌人数与频繁生成/回收场景下，系统应避免明显掉帧与卡顿。为满足该目标，项目采用`GameObjectPoolManager`池化复用、预加载与批量回收方案，减少Instantiate/Destroy抖动和GC压力。

（2）可维护性需求。项目要求“新增内容不破坏旧逻辑”。因此，核心机制采用模块分工和单一职责设计：例如战斗执行在`WeaponController`，武器槽管理在`WeaponManager`，难度计算在`DifficultySystem`，配置定义在ScriptableObject，避免“一个脚本管全部”。

（3）可配置与可扩展需求。难度、商店、刷怪、掉落等平衡参数应由配置和策略资产控制，而非硬编码散落在业务脚本。当前工程已将敌人缩放、商品价格、刷新价格、商品抽取、购买规则收敛到`Core/ValueSystem`策略体系，支持快速调参与A/B对比。

（4）鲁棒性需求。面对依赖晚到、异步加载失败、跨场景对象失效等情况，系统应尽量降级而非崩溃。项目通过`GameContext`注册查找、协程超时、空值判断、状态守卫等手段提升运行稳定性。

2.2 核心玩法循环建模
本项目的核心循环可建模为：“波次战斗 -> 波次结算 -> 升级选择 -> 商店决策 -> 下一波战斗”。其中，战斗阶段主导资源产出，波间阶段主导资源转换，二者交替构成完整体验。该循环的关键不在单次战斗输赢，而在连续多波中的“成长速度”与“敌人增长速度”是否匹配。

从资源视角看，系统至少包含五类关键变量：生命值、经验值、金币、武器槽位、道具效果。生命值是生存约束，经验值驱动升级机会，金币决定商店选择强度，武器槽位限制构筑上限，道具效果影响属性结构。上述变量并非独立变化，而是通过状态事件相互耦合：击杀触发经验与掉落，升级触发面板，面板选择反向改变属性并影响后续击杀效率，最终形成闭环。

从节奏视角看，系统采用“波内升压、波间喘息”的双相结构。`WaveConfig`中的`spawnIntensityCurve`控制同一波内部强度增长，`ModeConfig`与`DifficultySystem`控制跨波倍率增长，无尽模式在通关后继续叠加指数增长。该设计使前期学习成本较低，中期形成策略压力，后期提供挑战上限，符合Roguelike重复游玩的节奏预期。核心循环关系可在图2-2中给出状态化表达。

2.3 总体架构设计
2.3.1 架构分层
为降低复杂度并提升可维护性，系统采用五层架构（见图2-1）：

第一层为表现层，包含UI面板、动画、音效与可视反馈，负责“玩家看见什么”。`StorePanel`、`UpgradePanel`、`GamePanel`等面板属于该层。

第二层为业务层，负责“规则如何执行”。`WaveManager`管理波次生命周期，`SpawnManager`负责生成，`WeaponController`负责攻击结算，`InventoryManager`负责道具持有与应用。

第三层为数据层，采用ScriptableObject组织静态配置，负责“系统依据什么参数运行”。`ModeConfig`、`WaveConfig`、`ItemConfig`、`EnemyConfig`、`WeaponConfig`构成主要数据资产。

第四层为策略层（ValueSystem），负责“公式在哪里定义”。敌人缩放、商店价格、刷新价格、抽取概率、购买规则均通过策略SO实现，业务层只传上下文并读取结果，避免业务代码中散落公式。

第五层为运行支撑层，负责“系统如何稳定运行”。`GameSessionManager`处理跨场景会话，`GameContext`处理服务注册与依赖查找，`GameObjectPoolManager`处理对象复用与统一回收。

2.3.2 关键对象关系
在对象关系上，`GameMgr`处于局流程主控位置，负责初始化玩家、切换游戏状态并广播状态事件；`GameContext`作为场景内服务收口点，向`WaveManager`、`SpawnManager`、UI等模块提供依赖查询；`GameSessionManager`负责模式、角色、武器等会话数据跨场景保存；`WaveManager`与`SpawnManager`共同驱动战斗阶段，并通过UI面板衔接波间决策。

该关系设计的核心价值是把“谁拥有状态”与“谁消费状态”分离：状态主控尽量集中，业务执行模块只关注本职责，从而减少循环引用和隐藏耦合。图2-3建议以时序方式展示初始化链路，明确`GameMgr -> GameContext -> Player组件 -> WaveManager`的依赖就位顺序。

2.4 关键流程设计
2.4.1 开局初始化流程
开局流程由`GameMgr`发起。系统先完成Addressables初始化，再获取`GameSession`，随后加载`GlobalStatConfig`与`StatClampConfig`。配置就绪后实例化玩家对象，补全关键组件，设置角色外观、注入属性配置、初始化背包道具，并将玩家注册进`GameContext`。当玩家与核心服务全部就绪后，`GameMgr`将状态切换为`Playing`，触发`WaveManager`开始波次初始化。

该流程的关键在于异步步骤顺序控制和失败分支处理：任一关键配置缺失时流程应中止，避免“半初始化状态”进入战斗。图2-3可直接展示各对象消息调用顺序，便于答辩时解释鲁棒性设计。

2.4.2 单波执行流程
单波开始时，`WaveManager`读取当前`WaveData`，设置倒计时、生成点数量、强度曲线参数，并启动刷怪协程。刷怪过程中，普通敌人按权重池抽取并按锚点簇生分配，特殊敌人按延迟与数量独立投放。`SpawnManager`负责预警特效、敌人实例生成与活跃列表维护，`Enemy`子类负责行为执行与伤害交互。

当波次计时归零时，`WaveManager`终止刷怪、清理场景借出池对象、关闭战斗面板并进入波间逻辑。该流程重点解决“协程残留导致越波生成”的问题，项目通过`spawnSessionId`实现会话级失效判断，保证旧协程在波次切换后不再生效。图2-4建议用活动图表示这一生命周期。

2.4.3 波间商店/升级流程
波间阶段首先处理本波累计升级次数，逐次弹出`UpgradePanel`完成选择；升级流程结束后进入`StorePanel`，玩家可执行购买、刷新、锁定、出售、合成操作。所有购买行为均需经过策略规则评估，并与金币校验和库存状态联动，确保经济闭环正确。

流程结束由“继续”按钮触发：关闭商店、恢复`Playing`状态、回血并进入下一波。该过程是玩家“将资源转化为战力”的核心环节，也是构筑差异产生的主要来源。

2.4.4 胜利/失败与无尽分支流程
当普通波次全部完成后，系统进入终局选择：玩家可选择普通胜利或继续无尽。若进入无尽，`WaveManager`标记无尽状态并复用最后波模板叠加倍率增长；若玩家在无尽中死亡，结算展示可按设计映射为“无尽成绩型胜利展示”。若在普通流程死亡，则直接进入失败结算并清理局内金币。

此分支设计使“通关后继续挑战”与“结算呈现逻辑”解耦，既保留无尽模式目标感，也保持流程代码清晰。

2.5 数据驱动与配置规范
项目采用“配置定义内容、策略定义公式、业务执行流程”的数据驱动原则。为保证多人协作与后续调参效率，配置需满足可读、可校验、可追踪三项要求。

（1）`ModeConfig`规范。用于定义模式级参数，包括敌人倍率、经济倍率、治疗削减、无尽增长以及商店策略入口。该配置不直接进行数学计算，只提供策略所需参数。

（2）`WaveConfig`规范。用于定义波次结构，包含波时长、生成点数量、普通怪权重池、特殊怪延迟池、强度曲线与场上上限。编辑时应保证每波至少存在可生成敌人，且关键字段符合最小值约束。

（3）`ItemConfig + EffectSO`规范。`ItemConfig`只负责展示和效果引用，不承载执行逻辑；属性改动由`StatEffectSO`负责，动态功能由`LogicEffectSO`负责。效果必须实现成对生命周期（Initialize/Remove），避免残留状态。

（4）`WeaponConfig`与敌人配置规范。武器数据应按等级组织并支持运行时变体；敌人配置中攻击距离、速度、伤害等基础值需与动画参数一致，防止“配置有效但表现异常”。

（5）配置校验与版本管理。建议在开发流程中为关键配置建立检查清单（见表2-3），包括字段非空、数值范围、引用完整性、策略绑定完整性。每次数值改动记录版本号和改动理由，便于实验回溯。

通过上述规范，系统可在不修改核心业务脚本的前提下完成大多数平衡调整和内容扩展，为后续第4章实验与第5章优化提供可复现实验基础。

2.6 本章小结
本章从需求、循环、架构、流程、配置五个维度建立了项目的总体设计模型。需求层明确了“功能完备 + 工程稳定 + 性能可控”的目标；架构层通过分层与依赖收口降低了系统耦合；流程层保证了开局、战斗、波间、结算四类关键路径的可追踪执行；数据层通过ScriptableObject与策略SO完成可配置化落地。上述设计为后续“核心系统实现”提供了统一约束和评价基准。

第3章 核心系统设计与实现

3.1 游戏会话与状态管理系统

3.1.1 GameSessionManager跨场景会话管理

在个人项目开发中，我最先遇到的问题是“场景切换后选角、选武器、选难度信息丢失”。如果每个场景都重新手工传参，不仅代码重复，而且极易出现初始化时序错误。为此，我实现了`GameSessionManager`作为全局单例会话管理器，负责加载并持有`GameSession`对象，并通过`DontDestroyOnLoad`在场景间保持有效。该管理器在`Awake`阶段自动触发会话加载，提供同步读取接口`GetSession()`与异步等待接口`GetSessionAsync()`。这种设计可确保`GameMgr`在真正拿到会话后再创建玩家，避免“模式为空却进入战斗”的非法路径。

从工程角度看，会话层的意义不仅是“存数据”，更是“定义本局的输入边界”。在我当前的实现中，会话至少包含选中角色、选中武器和选中模式三个关键项。其后所有运行时逻辑（例如敌人倍率、商店策略、初始道具）都依赖这三个输入。换言之，会话是局内系统的起点，不应在中途被业务逻辑随意覆盖。这也解释了我在实现中保留了`SetSelectedCharacter`、`SetSelectedWeapon`、`SetSelectedMode`专用入口，而没有直接暴露对象内部字段给任意脚本写入。

3.1.2 GameMgr状态控制与时间尺度同步

`GameMgr`是当前项目的流程主控。其核心职责是：初始化全局配置与玩家、控制游戏状态流转、广播状态事件、处理终局展示。状态枚举包括`Waiting`、`Playing`、`Paused`、`Victory`、`Defeat`。与一般实现不同，我将状态切换与`Time.timeScale`同步封装在同一逻辑中，确保暂停与结算阶段不会出现“UI显示暂停但游戏仍在运行”的不一致问题。

在战斗流程中，`GameMgr`并不直接管理每一波刷怪细节，而是通过状态事件驱动`WaveManager`进入对应工作模式。这样可以避免主控脚本“越权”到战斗细节层，保持职责清晰。以波间流程为例：当`WaveManager`需要玩家进行商店/升级决策时，将状态切换为`Paused`；当玩家完成选择，`StorePanel`回调恢复`Playing`。整个链路中，`GameMgr`只负责状态合法性与时间尺度一致性，具体业务由各模块完成。

另外，我在`GameMgr`中加入了`GameEndPresentationKind`来区分“逻辑状态”与“结算展示类型”。这在无尽模式中尤为重要：玩家在无尽中死亡，逻辑上是`Defeat`，但展示上可定义为“无尽成绩结算”。这种二元拆分避免了大量`if-else`分支侵入到UI层，是我在后期重构中获得的关键经验。

3.1.3 GameContext依赖注册与服务定位

在Unity项目中，最常见的维护问题之一是“到处GetComponent和Find导致耦合失控”。为降低依赖分散，我实现了`GameContext`作为局内服务收口模块。`GameMgr`、`WaveManager`、`SpawnManager`和玩家实例会在就绪后显式向`GameContext`注册。其他模块统一通过`TryGetXxx`方法获取依赖，而不是自行全局查找。

这套机制带来的收益主要有三点。第一，依赖关系可读。读`GameContext`即可知道系统有哪些关键服务对象。第二，容错能力更好。`TryGet`接口天然支持“暂未就绪”场景，业务脚本可以选择等待或降级处理。第三，测试更方便。调试时我可以集中检查`GameContext`注册状态，快速定位“为什么某模块拿不到引用”。

当然，服务定位并非银弹。若所有脚本都无约束地访问`GameContext`，仍会形成隐式耦合。因此在实现中，我坚持“谁需要谁才取，不做跨层调用”。例如UI层不会直接操作敌人管理器，而是通过`WaveManager`暴露的流程入口间接驱动。

3.2 波次与刷怪系统

3.2.1 WaveConfig数据结构设计

波次系统的核心不只是“刷怪数量”，而是“时间结构 + 敌人结构 + 强度结构”的统一表达。为此，我将波次参数集中在`WaveConfig`中，单波由`WaveData`描述。关键字段包括：`waveDuration`、`waveStartDelay`、`normalEnemySpawns`、`specialEnemySpawns`、`spawnPointCount`、`baseSpawnInterval`、`minSpawnInterval`、`minSpawnCountPerTick`、`maxSpawnCountPerTick`、`spawnIntensityCurve`、`maxAliveEnemies`等。

这种设计有两个优点。其一，关卡策划逻辑可视化。通过Inspector即可观察一波内“何时快、何时慢、怪物怎么混”。其二，运行时计算清晰。`WaveManager`只需读取配置并按当前进度计算“当前间隔与当前批量”，不需要硬编码多套阶段规则。特别是`spawnIntensityCurve`的引入，使“同一波内部强度爬坡”可以通过曲线资产直观调节，而非写死分段函数。

普通怪采用加权池结构`NormalEnemySpawnData`，权重为0时即不参与抽取。特殊怪采用`SpecialEnemySpawnData`，支持固定数量与延迟生成。该组合可同时满足“持续压力”和“事件峰值”两类设计目标。

3.2.2 WaveManager波次调度与中断控制

`WaveManager`是战斗节奏中枢。其主要职责包括：初始化当前模式与波次配置、预加载敌人资源、开启与结束波次、维护计时、驱动刷怪协程、处理波间面板、触发终局分支。在代码层，我将其状态控制拆解为多个布尔标记（如`isGamePlaying`、`isWavePaused`、`isWaitingForPlayerChoice`、`isSpawning`）与多个协程引用（初始化、等待就绪、刷怪、波间流程），并提供统一停止入口`StopManagedCoroutines`。

为了防止协程跨波污染，我实现了`spawnSessionId`机制：每次波次结束自增，刷怪协程在关键节点检查会话ID是否仍有效。这样即便某个协程未及时停止，也会在下一次检查时自动失效，不会把旧波次生成任务带入新波次。该机制在高频暂停/恢复测试中显著提高了流程稳定性。

在流程逻辑上，`WaveManager`将“战斗阶段”和“决策阶段”明确隔离。战斗阶段只做计时与刷怪；阶段结束后先清场、隐藏战斗HUD，再按本波累计升级次数弹出升级面板，最后进入商店。这种顺序保证玩家每波都经历“战斗结算 -> 成长选择 -> 经济决策”的固定节奏，避免系统之间抢占UI焦点。

3.2.3 SpawnManager预加载、生成点与预警机制

`SpawnManager`负责敌人资源预加载、生成点计算、实例生成和活跃列表维护。为减少运行时卡顿，我在开局时根据全部波次配置收集敌人键并预加载到缓存字典；生成时优先取缓存预制体，再走对象池实例化。该流程可显著降低首次遇怪时的卡顿风险。

生成点逻辑采用地图矩形区域随机采样。每波可配置生成点数量，实际每次刷怪会从当前生成点中抽锚点，并在锚点周围按`burstRadius`簇生，形成“团簇涌入”效果。相比完全均匀随机，这种方式更接近类幸存者游戏常见的压迫感分布。

预警机制通过`SpawnEnemyWithTelegraph`协程实现：若启用预警，则先生成地面警示特效，等待指定时长后再生成敌人。协程内部支持`shouldContinueSpawning`回调，确保波次已结束时可中止预警并回收对象，避免“波次结束后仍落怪”。

3.2.4 同屏上限、强度曲线与无尽衔接

为了控制性能与难度，我在每次刷怪前使用`maxAliveEnemies`约束可生成数量。即便当前tick理论应刷更多怪，也必须先考虑场上存活数。这个限制一方面是性能保护阀，另一方面也防止极端参数导致战斗完全不可读。

强度曲线方面，`WaveManager`通过当前波进度`t`评估`spawnIntensityCurve`，并用线性插值映射到“刷新间隔”和“每批数量”。随着`t`上升，间隔缩短、批量增大，实现波内渐进升压。跨波难度则交由`DifficultySystem`与策略资产统一处理。

无尽模式衔接采用“复用最后一波模板 + 模式倍率增长”的方式。这样我无需维护另一套无尽配置，也能保证玩家在已熟悉的战斗结构上承受持续增长压力。实践表明，该方案在个人项目中成本低、稳定性高、可调节性强。

3.3 敌人系统与轻量AI实现

3.3.1 Enemy抽象基类设计

我将敌人共性行为统一到`Enemy`抽象基类，包括：生成初始化、玩家查找、难度倍率应用、受伤与死亡、经验/金币掉落、击退接口、地图边界约束和分离力计算。此设计避免了重复代码，同时保证不同敌人在“结算规则层”一致。

`InitializeForSpawn`是基类的关键入口。无论是首次实例化还是对象池复用，都必须走该方法重置运行时状态。这样可有效避免对象池常见问题，例如残留速度、残留动画状态、残留目标引用。基类还提供`IsWakeReady`机制，为生成后短暂僵直预留窗口，防止敌人刚出现即贴脸攻击。

另外，我在基类中统一实现`ComputeSeparationForce`。该方法基于邻域检测计算排斥向量，使敌人群体更自然地形成围攻，而非完全重叠。对类幸存者游戏而言，这种“略松散包围”通常比精确阵形更符合实际体验目标。

3.3.2 EnemyMelee近战行为实现

`EnemyMelee`采用“追击—进入攻击圈—动画命中帧结算”的行为链。移动阶段由追踪力与分离力合成，接近玩家后降低速度并准备攻击；当满足攻击范围与冷却条件时进入攻击动画，命中由动画事件`OnAttackHit`触发，结束由`OnAttackAnimEnd`解锁。该机制保证了视觉动作与伤害结算的一致性。

为提高战斗手感，近战敌人实现了持续时间型击退：在击退窗口内线性插值移动并暂时停止追击。相比瞬间位移，时序击退在视觉上更自然，也便于玩家理解控制效果。

3.3.3 EnemyHarpoonFish双圈层行为实现

远程敌人`EnemyHarpoonFish`采用双圈层模型：兴奋圈外随机游走，兴奋圈内朝玩家加速追击，进入投掷圈后停下并按冷却投掷鱼叉。该模型在行为上既保留“远程单位机动性”，又避免全程站桩，能提升战斗读图性。

投掷逻辑采用“起手锁定方向，释放帧按锁定方向发射”的方式。这样可避免动画期间因目标位移导致朝向抖动，也更符合玩家对“敌人已瞄准再出手”的直觉。投掷物由对象池生成，并在组件初始化时写入伤害、速度、最大飞行距离等参数。

3.3.4 难度倍率一致性应用

敌人最终属性不直接读取配置基础值，而是先由`DifficultySystem`给出当前波倍率，再在基类统一应用到生命、伤害、速度和攻击冷却。此方案保证了不同敌人在同一难度下增幅规则一致，减少“某类敌人被遗漏”导致的平衡偏差。

3.4 玩家成长与战斗系统

3.4.1 移动、属性与事件链

玩家移动由`PlayerMovement`实现，读取轴输入后通过刚体速度推进，并通过`PlayerEvents`广播移动与停止事件。虽然该逻辑简单，但其价值在于把输入与表现解耦：动画、音效、UI可订阅事件而不需侵入移动脚本。

属性系统`StatHandler`采用统一公式：最终值 = 基础值 + 固定修正 + 基础值×百分比修正，并在计算后按`StatClampConfig`进行上下限钳制。该设计保证属性增长可控，避免数值爆炸。基础值由全局配置与角色初始修正叠加得到，后续道具效果通过修正字典动态增减。

3.4.2 经验与等级推进

经验链路由`ExperienceManager`和`LevelSystem`协作完成。击杀敌人后`Enemy`通过`PlayerEvents`触发经验增加；`ExperienceManager`累计经验并判断是否可连续升级；`LevelSystem`负责等级上升与升级事件触发。该分工将“经验计算”与“等级状态”拆开，增强可测试性。

我在实现中保留了配置加载等待逻辑，确保`LevelConfig`未就绪时不会误触发升级。该处理对Addressables异步加载项目尤为重要。

3.4.3 金币与局内经济基础

`CoinManager`负责金币增加、扣除、清零和广播。商店购买、刷新、出售都通过该模块进行余额校验与扣款，避免多个系统各自维护金币状态造成不一致。局末清理逻辑由`GameMgr`触发，保证新局不会继承旧局余额。

3.5 武器系统设计与实现

3.5.1 WeaponManager武器槽与挂载位

我将武器持有与显示管理集中在`WeaponManager`。系统定义了7个挂载位：6个环形普通位与1个单武器位。开局仅持有一把武器时，优先放在单武器位；当武器数增加时自动迁移到普通位。这种设计兼顾了“单武器时视觉居中”和“多武器时布局均衡”。

`WeaponManager`还负责武器添加、移除、快速合成、手动合成和出售后的槽位压紧。通过事件回调（添加/移除/合成/数量变化），UI和控制层可及时同步状态。

3.5.2 WeaponController攻击执行

`WeaponController`承担攻击调度和伤害结算。它在每个挂载位维护独立冷却时间，并结合自动瞄准选取当前范围内最近敌人。近战与远程分开处理：近战依赖命中盒触发，远程生成投射物并初始化飞行参数。

为避免近战挥砍期间朝向跳变，我加入了“挥砍锁定旋转”机制：攻击触发后锁定挂载位旋转，动画结束事件再解锁。这一细节明显提升了近战命中的一致性和视觉稳定性。

3.5.3 合成与经济联动

当武器槽已满时，系统优先判断是否可与新武器合成。合成条件为同名、同类型、同等级且未满级。若可合成则替换为更高等级运行时副本；若不可合成则拒绝购买。出售价格通过商店价格策略统一计算，避免买卖体系割裂。

该机制使“背包容量限制”从纯约束转化为策略玩法：玩家可通过合成腾出空间，也可出售低价值武器换取中短期经济优势。

3.6 道具效果系统

3.6.1 容器与执行分离

在道具系统中，我明确区分“道具配置容器”和“效果执行逻辑”。`ItemConfig`仅保存展示信息、品阶、价格、效果列表，不直接负责运行时行为。实际行为由`EffectSO`子类承担。这种分离可让同一效果复用于不同道具组合，减少重复定义。

3.6.2 属性型效果与逻辑型效果

`StatEffectSO`通过`StatModifier`列表对`StatHandler`施加固定或百分比修正，适合攻速、伤害、移速等数值变化。`LogicEffectSO`则通过反射动态挂载组件实现机制型效果，例如额外行为、触发器、特殊被动。两类效果都遵循`Initialize/Remove`生命周期，保证可逆。

`LogicEffectSO`支持`Single`与`Stackable`两种堆叠方式。前者同类效果只保留一份，后者允许多实例并行。这使我可以根据设计意图控制机制叠加上限。

3.6.3 EffectManager堆叠与清理

`EffectManager`维护效果引用计数与逻辑组件映射。添加道具时按层应用，移除道具时按层回退，最后一层移除时才真正卸载效果。该机制可避免“重复获得同道具后移除一次就全部失效”的常见错误。对象销毁时调用`RemoveAllEffects`做兜底清理，避免动态组件残留。

3.7 商店与升级系统

3.7.1 StorePanel交互流程

`StorePanel`在波间显示，负责商品刷新、锁定、购买、继续战斗。商品槽位固定为4个，刷新时优先保留锁定槽，并将锁定项前置，未锁定项重新抽取。该机制给予玩家“保留关键商品”的策略空间，同时保留随机性。

面板显示时会重新绑定场景依赖，避免持久化UI在跨局后引用失效。金币变化、武器背包变化、属性变化均在面板内联动刷新，确保玩家决策依据实时有效。

3.7.2 购买决策与规则校验

购买行为不是简单的“钱够就买”。我将可购买性判定下沉到`ShopPurchaseRuleStrategy`，由策略返回`ShopPurchaseDecision`（是否可买、实际价格、失败原因）。这样业务层只执行流程，不承担规则判断细节。对个人项目而言，这种结构特别有助于后期反复改规则而不破坏UI流程。

3.7.3 UpgradePanel升级选择

升级面板每次提供3个道具选项，支持按玩家等级进行权重偏置。当前实现中，等级越高，高等级道具权重越大。虽然该逻辑仍可继续精细化，但已能在原型阶段形成“成长推进越深，候选质量越高”的体验趋势。

3.7.4 商店、升级与波次衔接

在流程顺序上，我坚持“先升级后商店”。原因是升级通常影响基础属性，商店则偏向资源投入决策。先升级可让玩家在进入商店时基于最新属性做更精确选择。该顺序在多轮测试中表现出更好的决策可读性。

3.8 难度与经济策略系统

3.8.1 DifficultySystem统一入口

`DifficultySystem`在每局创建，在局末关闭，按当前波次和无尽状态计算敌人倍率、刷怪倍率、材料掉率等结果。业务层不直接写倍率公式，只调用系统接口获取结果。该做法显著降低了多模块重复计算导致的偏差。

3.8.2 价格、刷新、抽卡与购买策略

商店相关策略分别由以下模块承担：`ShopProductPriceStrategy`计算商品最终价格，`ShopRefreshPriceStrategy`计算当前刷新花费，`ShopProductRollStrategy`决定随机商品，`ShopPurchaseRuleStrategy`决定购买是否合法。四类策略互相独立但共享模式上下文，便于单项调参与实验对比。

以价格策略为例，我采用“基础价 × 模式倍率 × 波次线性/二次增长”结构。这样前期增长温和、后期增长明显，能够在不提高操作难度的情况下提高经济压力。抽卡策略则用等级权重曲线控制不同品阶的出货概率，保证中后期成长预期。

3.8.3 策略化设计的工程收益

在开发后期，我多次调整刷新费用和高阶商品出现概率。由于公式集中在策略资产中，我几乎不需要修改UI或商店流程脚本。该实践验证了策略化设计在独立项目中的现实价值：调参成本低、风险隔离好、可回滚性强。

本章围绕项目核心系统给出了从架构到实现的完整论述。通过状态与会话管理，我保证了流程一致性；通过波次与敌人系统，我构建了可配置的战斗压力模型；通过玩家成长、武器道具、商店升级和策略系统，我形成了可迭代的构筑与经济闭环。整体上，系统已满足“可玩、可调、可扩展”的阶段性目标，为后续实验与优化提供了可靠基础。

4.1 实验目标与评价指标

（4）战斗效率指标：每分钟击杀数、每分钟受击次数、平均DPS估算、场上敌人峰值。

4.3 参数组设计

4.3.1 波次参数组

我重点调整`baseSpawnInterval`、`minSpawnInterval`、`spawnIntensityCurve`和`maxAliveEnemies`。其中，间隔参数决定敌人注入速率，上限参数决定场上压力峰值，强度曲线决定同一波内压力爬坡形态。调参目标是避免“前期过空”与“后期瞬间爆表”。

敌人倍率通过`EnemyScalingStrategy`统一输出。我主要对生命和伤害增长进行了组合测试，观察“打不动”与“被秒杀”两类失败体验的出现区间。最终版本中，生命增长略高于伤害增长，以保留后期操作容错空间。

4.3.4 升级与经验参数组

升级节奏通过`LevelConfig`和经验奖励共同决定。我通过控制“平均每波升级次数”来稳定成长反馈，避免连续无升级导致体验停滞，也避免升级过密造成属性失控。

4.4.1 难度曲线结果

在最终参数下，普通模式前期（1~3波）玩家死亡率较低，主要用于建立操作与构筑认知；中期（4~7波）死亡率明显上升，体现策略压力；后期及无尽阶段，生存表现与构筑质量关联增强，表现出较好区分度。整体曲线符合“学习期-挑战期-极限期”的目标。

4.4.2 成长与经济结果

第5章 性能优化与工程化验证

5.1 性能问题定位

项目早期的主要性能问题集中在三处：第一，敌人和投射物频繁创建销毁导致GC抖动；第二，波次切换后场景残留对象影响下一波统计；第三，首次加载某些资源时出现瞬时卡顿。针对这些问题，我使用Unity Profiler定位CPU热点与GC分配来源，并结合代码路径逐项优化。

5.2 对象池优化

`GameObjectPoolManager`是性能优化的核心模块。其实现要点包括：按Prefab维护队列缓存、实例到预制体映射、单类与全局缓存上限、延迟回收、全量借出实例回收。波次结束时，`WaveManager`调用统一回收接口清理活跃借出对象，避免残留实例继续参与逻辑。

该方案直接减少了Instantiate/Destroy调用频率，也降低了内存抖动。实践中，战斗高峰期帧时间波动明显收敛，GC尖峰出现频率降低。

5.3 资源加载与生命周期管理

我在`SpawnManager`中加入敌人预加载流程：进入战斗前根据所有波次收集敌人配置并批量加载预制体，运行时仅从缓存取资源。商店和UI相关预制体也采用预加载与销毁释放策略，防止跨局资源泄露。

另外，针对协程和事件监听，我在对象禁用或销毁时统一解绑与停止，降低悬挂引用风险。这一工程化细节对长期运行稳定性有直接影响。

5.4 优化效果评估

在同等压力场景下，优化后版本表现出以下趋势：平均帧率提升，低帧占比下降，GC Alloc峰值降低，波次切换稳定性提升。虽然项目仍未达到商业化深度优化水平，但在个人项目规模下已满足“可持续游玩与调试”的目标。

第6章 系统测试与结果评估

6.1 测试策略

我采用“功能正确性优先、稳定性并行验证”的测试策略，将测试分为功能测试、边界测试、异常测试与压力测试四类。每类测试都尽量给出可重复步骤与可判断结果，避免主观描述。

6.2 功能测试结果

功能测试覆盖开局初始化、波次推进、升级面板、商店刷新购买、武器合成出售、终局结算与无尽分支等主路径。结果表明，核心流程可稳定闭环，关键交互能按预期触发。少量问题主要集中在极端时序下的UI显示同步与个别配置缺失容错。

6.3 边界与异常测试

边界测试包括：空商品池、空敌人池、配置缺失、余额不足、连续快速点击、暂停恢复频繁切换等。异常测试关注异步加载失败与依赖晚到场景。通过`TryGet`容错、超时等待和状态守卫，系统在多数异常下能安全退出或降级，不会造成整体崩溃。

6.4 压力测试

压力测试主要提升同屏敌人数、缩短刷怪间隔、增加投射物频率。测试结果显示，在当前优化方案下，系统可维持可玩帧率并保持流程正确。瓶颈仍主要出现在极限密度下的物理与碰撞开销，后续可通过进一步分层碰撞与更新节流优化。

第7章 总结与展望

7.1 工作总结

本文围绕“基于Unity3D的Roguelike类PC游戏设计与实现”完成了从需求到实现、从调优到验证的完整实践。通过构建状态管理、波次刷怪、敌人行为、成长构筑、商店升级、策略平衡和性能优化等模块，我实现了一个可持续迭代的游戏原型框架，并验证了其在个人开发规模下的工程可行性。

7.2 主要贡献

第一，我提出并落地了“配置驱动 + 策略解耦 + 组件协作”的实现路径，使玩法逻辑与平衡公式解耦，提高了维护效率。第二，我构建了围绕波次流程的完整系统闭环，打通了战斗、成长、经济与结算。第三，我在对象池与资源生命周期管理方面进行了工程化实践，改善了高同屏场景下的运行稳定性。第四，我通过实验指标而非纯体感完成参数调优，使结论具备一定可解释性。

7.3 局限性分析

尽管当前项目已满足原型目标，但仍存在局限。其一，敌人行为复杂度仍较低，尚未引入更丰富的协同行为与机制交互。其二，地图与环境机制较弱，战斗主要依赖单位对抗。其三，自动化测试与数据埋点能力不足，调优仍依赖较多人工流程。其四，长期Meta进度系统尚未建立，重玩动机主要来自局内构筑变化。

7.4 后续工作

针对上述局限，我后续计划包括：第一，引入更高层敌人行为模型（分层状态机或行为树），增强中后期战斗变化。第二，扩展地图事件与环境机制，提升局内策略维度。第三，构建更系统的数据埋点与可视化分析工具，形成半自动平衡流程。第四，设计长期Meta系统（角色成长线、解锁树、成就路径），增强跨局目标感。第五，完善工程质量体系，包括自动回归脚本、配置校验工具和发布流程规范。
